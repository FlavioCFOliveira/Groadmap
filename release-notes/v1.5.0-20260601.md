# Groadmap v1.5.0 — Release Notes

**Version:** 1.5.0
**Date:** 2026-06-01
**Type:** Minor (backward-compatible new functionality)
**Comparison:** https://github.com/FlavioCFOliveira/Groadmap/compare/v1.4.0...v1.5.0

---

## Executive Summary

Groadmap v1.5.0 ships two major features. First, the new `rmp graph` command
gives every roadmap a Cypher-queryable knowledge graph backed by the GoGraph
engine, stored durably under `~/.roadmaps/<name>/graph/`. Five guard-railed
subcommands (`create`, `query`, `update`, `delete`, `search`) let users record
and traverse structural knowledge about their projects. Second, the storage
layout migrates from a flat-file model (`~/.roadmaps/<name>.db`) to a
per-roadmap home directory (`~/.roadmaps/<name>/project.db`), paving the way
for future per-roadmap artefacts. Both changes are backward compatible: existing
databases are migrated automatically at startup with no data loss.

## Development Items

### 1. `rmp graph` Command — Cypher-Queryable Knowledge Graph

**What:** A new top-level command family with five subcommands:

| Subcommand | Operation | Accepted Cypher Clauses |
|------------|-----------|------------------------|
| `graph create` | Add nodes/edges | `CREATE`, `MERGE` |
| `graph query` | Read | Read-only `MATCH ... RETURN` |
| `graph update` | Mutate existing | `SET`, `REMOVE` |
| `graph delete` | Remove | `DELETE`, `DETACH DELETE` |
| `graph search` | Read traversal | Read-only (including `[*N..M]`) |

**Why:** Tasks and sprints are structured records; specifications, modules,
decisions, and architectural dependencies are a graph. An agent or user
building a roadmap needs to record and query non-task knowledge — which spec
is implemented by which module, which module depends on which service, which
decisions block which specs. The GoGraph Cypher engine, already a transitive
dependency through the GoGraph module, provides the right storage primitive.

**Implementation:**
- `internal/commands/graph.go` (634 lines): handler functions for all five
  subcommands. Each handler opens the GoGraph WAL store at
  `~/.roadmaps/<name>/graph/`, executes exactly one Cypher query, commits,
  and closes before exit.
- `internal/commands/registry_graph.go` (245 lines): declarative registry
  entry for the `graph` family, including all subcommands, flags, examples,
  output shapes, and exit-code annotations.
- Guard-rail enforcement exits with code `6` when the query's operation class
  does not match the subcommand (e.g. passing a `CREATE` query to `graph query`).
- `--query` falls back to reading from standard input when absent. An empty or
  disconnected stdin exits with code `2`.
- JSON output covers all `expr.Value` types produced by GoGraph: `bool`,
  `int64`, `float64`, `string`, `[]any`, `map[string]any`, `time.Time` (RFC 3339
  UTC), `nil`, with a `fmt.Sprintf` fallback for unknown types.
- Read subcommands return `{"columns": [...], "rows": [[...], ...]}`.
- Write subcommands return `{"ok": true}` when no `RETURN` clause is present,
  or the full column/row shape when `RETURN` is present.

**Impact:** New command family registered in the declarative registry. No
existing commands are affected. The graph store is a new subdirectory
(`graph/`) under each roadmap home directory and is independent of the SQLite
database.

**SPEC:** `SPEC/GRAPH.md` (new, 417 lines) is the authoritative specification
for persistence layout, guard-rail rules, Cypher input precedence, output
schemas, exit codes, and security model.

### 2. Per-Roadmap Directory Layout and Automatic Legacy Migration

**What:** Each roadmap's SQLite database moves from
`~/.roadmaps/<name>.db` (a flat file in the data directory) to
`~/.roadmaps/<name>/project.db` (inside its own home directory, mode `0700`,
file mode `0600`). This layout was required before the graph store was
introduced, since the graph needs its own subdirectory under the same roadmap
root.

**Why:** A flat `<name>.db` file at the top level of `~/.roadmaps/` offered no
room for per-roadmap artefacts without another layout change later. The directory
model (`<name>/`) is the permanent container; the graph command is the first
consumer.

**Implementation:**
- `internal/utils/path.go`: `DBFileName`, `GetRoadmapDir`, `EnsureRoadmapDir`
  added; `GetRoadmapPath`, `RoadmapExists`, `ListRoadmaps` reworked for the new
  layout.
- `internal/utils/migrate.go` (new, 224 lines): `MigrateLegacyLayout` sweeps
  `~/.roadmaps/` on startup. For every regular file `<name>.db`, if
  `~/.roadmaps/<name>/project.db` does not exist, the file is atomically renamed
  (plus WAL/SHM sidecars, best-effort). Symbolic links, directories, and entries
  with invalid roadmap names are skipped silently. An existing `project.db` is
  never overwritten.
- `internal/db/connection.go`: `Open` calls `EnsureRoadmapDir` before opening
  the database.
- `internal/commands/roadmap.go`: `list` reports `<name>/project.db` paths;
  `remove` deletes the full `<name>/` home directory.
- The migration runs before command routing in `cmd/rmp/main.go`, skipped by
  `--help`, `--version`, and `--ai-help` exits.

**Impact:** Transparent to users. Any roadmap created under v1.4.0 or earlier
is migrated silently. No data loss is possible: the atomic rename either
succeeds or leaves the original file untouched. A warning is emitted to stderr
on the non-fatal conflict path (legacy file left untouched when a `project.db`
already exists at the destination).

### 3. AI Agent Contract Extensions

**What:** Three additions to the machine-readable contract emitted by
`rmp --ai-help`:

1. **`stdin_fallback` field** on each flag entry (`FlagEntry.StdinFallback *bool`,
   `omitempty`): signals that the subcommand reads the flag's value from stdin
   when the flag is absent. Used by all five `graph` subcommands for `--query`.
2. **`reads_stdin` field** on each subcommand entry (`SubcommandEntry.ReadsStdin
   *bool`, `omitempty`): signals that the subcommand may read from standard input.
3. **`build_knowledge_graph` workflow**: a four-step guide for populating and
   querying a roadmap's knowledge graph with Cypher, added to `staticWorkflows()`.
4. **`graph_guard_rail_mismatch` pitfall**: documents the exit-6 guard-rail
   behaviour and the correct subcommand selection.
5. **`graph_missing_query` pitfall**: documents the stdin-fallback behaviour and
   the failure mode when neither `--query` nor stdin is supplied.

**Why:** AI agents invoking `rmp graph` need to know: (a) that `--query` has a
stdin fallback so they can pipe Cypher correctly, and (b) that subcommand
selection is a strict guard rail that will fail with a specific exit code if
mismatched. Surfacing these in the contract prevents the most likely agent
mistakes without requiring the agent to read the full help text.

**Impact:** The contract JSON gains two new optional fields and additional
`common_workflows` and `pitfalls` entries. Existing contract consumers that
ignore unknown fields are unaffected. Consumers that process `pitfalls` and
`common_workflows` will receive the new entries.

### 4. Dependency Update: GoGraph Promoted to Direct Dependency

**What:** `go.mod` promotes `github.com/FlavioCFOliveira/GoGraph` from
`// indirect` to a direct `require` entry at
`v0.0.0-20260601121207-03162239610a`. The `go` directive is raised from
`1.25.0` to `1.26`.

**Why:** The graph command directly imports GoGraph packages (`cypher`, `expr`,
`store/wal`, `store/txn`, `store/recovery`). An indirect entry would be
misleading and could be removed by `go mod tidy`.

**Impact:** `go.sum` gains new hashes for GoGraph and its transitive
dependencies (ANTLR4, RoaringBitmap). Build time increases slightly on a cold
cache. Runtime binary size increases due to the GoGraph engine being linked in.

## Breaking Changes

None. All changes in this release are backward compatible with `v1.4.0`.

- The public CLI surface, exit codes, and existing JSON output schemas are
  unchanged.
- The storage layout migration is automatic and transparent.
- The `rmp graph` command is a pure addition with no effect on existing commands.

## Security Notes

- The legacy migration guards against symbolic-link traversal: only regular
  files are renamed; symbolic links (dangling, file-target, or directory-target)
  are skipped silently, preventing any rename or `chmod` outside
  `~/.roadmaps/`.
- The graph store directory is created with mode `0700` (owner-only), consistent
  with the existing roadmap home directory permissions.
- GoGraph Cypher queries are parsed and validated by the engine; Groadmap adds
  guard-rail enforcement at the CLI layer before handing off to the engine.
- No network access is introduced by this release.

## Known Issues

- No E2E test coverage for `rmp graph` subcommands. Tracked as a follow-up.
- SPEC-vs-code divergences inherited from v1.4.0 (unchanged):
  - `SPEC/ARCHITECTURE.md` maps `ErrInvalidInput` → exit `2`; implementation
    maps `ErrInvalidInput`, `ErrValidation`, `ErrFieldTooLarge` →
    `ExitInvalidData = 6`.
  - `SPEC/COMMANDS.md` `audit stats` JSON keys do not match implementation.
    Implementation behaviour is stable and adopted by tooling.

## Upgrade Instructions

1. Download or build the `v1.5.0` binary.
2. Replace the existing `rmp` binary in your `PATH`.
3. Run any `rmp` command (except `--help`, `--version`, `--ai-help`). If you
   have roadmaps created before `v1.5.0`, the storage layout migration runs
   automatically and silently: each `~/.roadmaps/<name>.db` is moved to
   `~/.roadmaps/<name>/project.db`.
4. Verify with `rmp roadmap list` — existing roadmaps appear as before.
5. No configuration changes are required. No data loss is possible.
