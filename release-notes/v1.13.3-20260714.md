# Release Notes — Groadmap v1.13.3

**Date:** 2026-07-14
**Type:** Patch (graph query correctness, database error classification, test integrity)
**Tag:** `v1.13.3`
**Previous release:** `v1.13.2` (2026-07-13)

---

## Executive Summary

Groadmap `v1.13.3` is a **correctness release**. Its headline is a fix in the
Cypher pattern evaluator of the embedded **GoGraph** knowledge-graph engine,
which moves from `v0.7.0` through `v0.8.0` to `v0.8.1`.

Up to GoGraph `v0.8.0`, pattern-predicate and pattern-comprehension
relationship-type matching inspected only the **first** stored relationship type
between an ordered pair of nodes. Over **parallel edges** — two or more
relationships of different types between the same ordered pair, which a
multigraph allows and which real knowledge graphs contain — every non-first type
was wrongly reported as absent.

This matters because `WHERE NOT (n)-[:TYPE]->()` is the canonical query shape for
finding **gaps** in a knowledge graph: a requirement with no test, a file with no
component, a feature with no specification. Under the old engine, those gap
audits returned **false gaps** — they invented missing relationships that were in
fact present. Anyone running completeness or coverage queries over a multigraph
was being misled.

This release also corrects a SQLite error-classification bug that could report a
non-uniqueness constraint failure as an "already in use" collision, and it
restores the integrity of two test suites that were not testing what they claimed
to test.

---

## Highlights

- **Graph gap audits now tell the truth.** `WHERE NOT (n)-[:TYPE]->()` no longer
  reports a relationship as missing when a parallel edge of that type exists.
  This reaches users through `rmp graph query`, `rmp graph search`, and the
  `rmp web` knowledge-graph query bar, which all run the same engine.
- **No silent misclassification of database errors.** A `CHECK`, `NOT NULL`, or
  `FOREIGN KEY` violation is no longer reported to the user as a uniqueness
  collision with exit code 5.
- **Two test suites that were not testing what they claimed now genuinely
  assert.** The database index tests exercised a lookalike schema and lookalike
  SQL; the model memory-layout benchmark logged the layout instead of asserting
  it. Both are now real, failing-on-regression tests.
- **Nothing to do on upgrade.** No on-disk graph store migration, no SQLite
  migration, no command or flag change.

## Changes by Category

### Fixed — Graph Query Correctness Over Parallel Edges (via GoGraph v0.8.1)

A **parallel edge** is two or more relationships of *different types* between the
same ordered pair of nodes — for example, a `Requirement` node that is both
`VERIFIED_BY` and `SUPERSEDED_BY` the same `Test` node. A labelled property graph
is a multigraph, so this is a legitimate and common shape.

The GoGraph pattern evaluator resolved the relationship type of such a pair by
taking the **first** stored type (`EdgeLabels(pair)[0]`). Every other parallel
type on that pair was therefore invisible to pattern matching. Given a graph
holding `(a)-[:FIRST]->(b)` and `(a)-[:SECOND]->(b)`:

| Query | Old (incorrect) result | New (correct) result |
|-------|------------------------|----------------------|
| `WHERE NOT (a)-[:SECOND]->()` | returns `a` — claims an edge that exists is missing | returns nothing |
| `WHERE (a)-[:SECOND]->()` | returns nothing | returns `a` |
| `type(r)` on a bound relationship variable | may report the first-stored type | reports the pattern-selected type |

The first-stored type always answered correctly, which is exactly why the defect
stayed hidden for so long: any query that happened to name the first type behaved
perfectly.

GoGraph `v0.8.1` moves the check into the single-relationship chokepoint
(`edgeMatchesRel`), which now tests **every** relationship type of the endpoint
pair. That fixes the outgoing, incoming, undirected, variable-length, and
comprehension paths at once. The pattern-comprehension fallback path now reuses
the canonical type resolver, so `type(r)` reports the pattern-selected type.

**Why users should care.** `WHERE NOT (n)-[:TYPE]->()` is how you audit a
knowledge graph for completeness. Under the old engine, a gap audit over a
multigraph produced false positives — relationships reported as missing that were
actually present — and the positive form of the same query silently omitted real
matches. Coverage and traceability conclusions drawn from those queries were
unsound.

### Fixed — Database Error Classification: Only Uniqueness Violations Are Collisions

`IsUniqueConstraintErr` decided whether a SQLite failure was a uniqueness
collision by masking the result code down to the **primary** code
`SQLITE_CONSTRAINT` (19). That primary code is shared by *every* constraint kind,
so it cannot tell a uniqueness collision apart from a `CHECK` (275), `NOT NULL`
(1299), or `FOREIGN KEY` violation.

The practical consequence was on the sprint insert and update paths: any
constraint failure on those statements was translated into `ErrAlreadyExists` and
reported to the user as `sprint order N is already in use`, with exit code 5
(`EXIT_EXISTS`) — even when the real failure had nothing to do with the execution
order.

The check is now made on the **extended** result codes, and only those:

- `SQLITE_CONSTRAINT_UNIQUE` (2067)
- `SQLITE_CONSTRAINT_PRIMARYKEY` (1555)

Exit code 5 keeps its documented meaning. It is simply no longer emitted for
failures that are not collisions; a non-uniqueness constraint failure now
surfaces as the failure it actually is.

### Changed — Dependency Refresh

| Module | From | To | Kind |
|--------|------|----|------|
| `github.com/FlavioCFOliveira/GoGraph` | `v0.7.0` | `v0.8.1` | Direct |

No other module changed in this release; `go.mod` and `go.sum` carry no other
delta.

The GoGraph upgrade is **API-additive and read-path only**. No exported
identifier was added, removed, or renamed; no Groadmap call site changed; and the
sole upstream production source change is in the Cypher pattern evaluator
(`cypher/pattern_eval.go`). It touches no write, commit, or recovery path, so
there is **no on-disk graph format migration**. An existing graph store opens and
queries unchanged.

Upstream reports both compliance invariants holding without regression at
`v0.8.1`: 100 % openCypher TCK compliance at the execution level
(3 897 / 3 897 scenarios) and 100 % ACID compliance.

### Changed — Wider Cypher DDL Grammar Upstream; Groadmap Still Rejects All DDL

GoGraph `v0.8.0` widened the Cypher DDL surface, accepting the modern
`CREATE CONSTRAINT ... FOR ... REQUIRE ...` grammar alongside the legacy
`ON ... ASSERT` form.

Groadmap's position is unchanged: **every `rmp graph` subcommand rejects all
DDL.** Because the upstream grammar grew, that contract is now pinned explicitly
by regression tests in `internal/cypherguard` covering both the modern and the
legacy grammar, including the `IF NOT EXISTS` variant. A widened upstream parser
cannot quietly open a DDL path through `rmp`.

### Internal

- **Database index tests now verify the real schema and the real queries.** The
  index tests constructed their own local schema and their own lookalike SQL.
  They could therefore pass while the production schema, or the production query,
  drifted away from the very index they claimed to exercise — a test that cannot
  fail for the reason it exists. Query assembly is now separated from query
  execution (`buildListTasksQuery`, `buildAuditEntriesQuery`, and the named
  `sprintTasksLookupQuery` constant), so the tests take the query plan of the
  **exact SQL production runs**, against the **real schema**. This is a
  behaviour-preserving refactor: the SQL and the bind arguments are unchanged.
  See `SPEC/DATABASE.md`, "Performance Optimization".
- **Model memory layout is asserted, not logged.** The struct-size check was a
  benchmark that printed the layout with `b.Logf` and asserted nothing, so a
  regression in the memory layout pinned by `SPEC/MODELS.md` could not fail the
  build. It is now a real test, `TestSpecifiedStructSizes`, which asserts the
  specified sizes and is scoped to 64-bit targets, as the specification states.
- **Knowledge-graph model documented.** `knowledge-model.md` is added as the
  authoritative description of the project knowledge graph's shape: the
  normalized node schema, the structural edges re-derived from the code, and the
  traceability and provenance contract.

## Breaking Changes

**None.** This release contains backward-compatible bug fixes, test-integrity
work, and documentation only.

No `rmp` command, flag, or JSON success schema is added, removed, or renamed. No
exit code changes meaning. The GoGraph upgrade is API-additive with no exported
API break and no on-disk store migration. The database schema version is
unchanged (`1.8.0`).

### A Note on Behaviour Changes and Semantic Versioning

Two behaviours *do* change, and both are corrections of defects rather than
contract changes:

1. **Graph pattern predicates now return different results** over parallel edges
   — because the previous results were wrong. A query that was silently lying now
   tells the truth. Code that depended on the old answers depended on a bug.
2. **A non-uniqueness constraint failure no longer yields exit code 5.** The
   meaning of exit code 5 is unchanged; it is simply no longer emitted for a
   failure that is not a collision. The previously emitted code was incorrect.

Under Semantic Versioning 2.0.0, a backward-compatible bug fix is precisely a
PATCH. Neither change adds, removes, or renames any part of Groadmap's declared
public API (commands, flags, JSON success schema, exit codes, database schema).

## Upgrade Notes

**No manual migration steps are required.** This release adds no SQLite migration
of its own, the database schema version is unchanged, and the GoGraph upgrade
needs no on-disk graph store migration. An existing graph store opens and queries
unchanged.

```sh
# Build from source (requires Go 1.26.5+)
git fetch --tags
git checkout v1.13.3
go build -o ./bin/rmp ./cmd/rmp

# Verify
./bin/rmp --version
# Groadmap version 1.13.3
```

If you maintain a knowledge graph that contains parallel edges, it is worth
**re-running any completeness or coverage audit** you previously trusted. Gap
queries of the form below may have reported false gaps under the old engine, and
will now return the correct answer:

```sh
# Find requirements that no test verifies.
# Under GoGraph v0.7.0/v0.8.0 this could report false gaps
# whenever a parallel edge was present on the same node pair.
rmp graph query -r myproject \
  --query "MATCH (r:Requirement) WHERE NOT (r)-[:VERIFIED_BY]->() RETURN r.key"
```

## Benchmark Evidence

This release fixes correctness defects, refreshes one dependency, and repairs
tests; it does not change the hot data path of any benchmarked `rmp` command. The
Groadmap CLI command benchmarks therefore remain the reference established in
`v1.3.0` and are not re-measured here.

The GoGraph upgrade is a correctness-only patch whose sole production change is in
the Cypher pattern evaluator, touching no traversal or execution hot path.
Upstream accordingly inherits its `v0.8.0` benchmark figures unchanged at
`v0.8.1`. No change in `rmp graph` latency was observed across the E2E suite.

The database change is a behaviour-preserving extraction of query assembly from
query execution: the SQL and the bind arguments executed in production are
byte-for-byte the same, so no query plan changes.
