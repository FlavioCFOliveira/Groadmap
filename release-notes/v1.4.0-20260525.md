# Groadmap v1.4.0

- **Version:** 1.4.0
- **Date:** 2026-05-25
- **Release type:** Minor (backward compatible)
- **Previous tag:** v1.3.0
- **Comparison:** https://github.com/FlavioCFOliveira/Groadmap/compare/v1.3.0...v1.4.0

## Executive summary

Groadmap v1.4.0 makes the CLI first-class for AI agents. It introduces the
**AI Agent Contract**, a machine-readable JSON description of every command,
flag, alias, enum, exit code, JSON output shape, canonical workflow and known
pitfall, exposed through `rmp --ai-help` and `rmp ai-help`. A set of
proactive discovery hints (a banner on every `--help` page, a stderr hint on
the error path, and an opt-in `AI_AGENT=1` environment-variable hint) ensures
that agents always find the contract entry point regardless of how they first
probe the binary. Internally, command dispatch is now driven by a declarative
command registry, so the contract is generated from the same source of truth
that routes runtime calls and cannot drift from the binary. The `sprint`
description length cap is raised from 500 to 2048 characters. No public CLI
contract changed; this release is a drop-in upgrade from `v1.3.0`.

## Highlights

### AI Agent Contract (`rmp --ai-help`)

A complete machine-readable contract is now produced by the binary:

```sh
rmp --ai-help            # full contract on stdout (JSON)
rmp ai-help              # equivalent command form
rmp task --ai-help       # scoped to the task family
rmp sprint create --ai-help   # scoped to a single subcommand
```

The emitted JSON describes:

- every command family, subcommand and alias;
- every flag, its type, default, and whether it is required;
- every enum value (status, priority, type);
- every exit code, with its semantics;
- the shape of the JSON output on success;
- a curated list of canonical workflows (sprint planning, reprioritising
  the backlog, closing a sprint, etc.);
- a curated list of common pitfalls (forgetting `-r`, mixing
  `sprint open-tasks` with `task list --status`, etc.).

The contract is generated from the same declarative registry that drives
runtime dispatch (`internal/commands/registry*.go`), which makes drift
between documentation and binary structurally impossible.

`--ai-help` takes precedence over `--help`, `--version`, `-r` and every
action flag. The scoping logic is independently unit-tested
(`cmd/rmp/aihelp_wiring_test.go`).

### Discovery hints for AI agents

Three complementary discovery surfaces ensure that any agent that touches
`rmp` finds the contract:

1. **Help banner**: every `--help` page (main help and every family or
   subcommand help) begins with a single-line banner pointing at
   `rmp --ai-help`. The banner string lives once, in
   `internal/commands/banner.go`, and is emitted through a small writer
   helper so it cannot drift.
2. **Error-path hint**: every error message is followed by
   `AI agents: run `+"`rmp --ai-help`"+` for a machine-readable command contract.`
   The hint is suppressed when the invocation itself is serving the
   contract (avoiding recursive noise).
3. **`AI_AGENT=1` env-var hint**: when the environment variable is set,
   the same hint is the first line written to stderr for the entire
   invocation. A `sync.Once`-guarded helper deduplicates the env-var
   path against the error path so the hint appears exactly once.

The suppression rules are part of the SPEC and are covered by both Go
unit tests and the Python E2E suite.

### Declarative command registry

`cmd/rmp/main.go` no longer routes commands through handcrafted switch
chains. A single declarative registry
(`internal/commands/registry.go` plus `registry_task.go`,
`registry_sprint.go`, `registry_data.go` and `registry_aihelp.go`)
holds the structure of every command family. The dispatcher and the
AI Agent Contract generator both read from this registry, so the
contract is, by construction, a faithful description of the runtime.

The refactor is internal: the public CLI surface is unchanged. Roughly
1,800 net lines were added (registry + tests) and roughly 350 were
removed from the per-family handler files.

### Sprint description cap raised to 2048 characters

The validator upper bound for the sprint `description` field was raised
from `500` to `2048` characters (`internal/models/consts.go`,
`SPEC/DATABASE.md`, `SPEC/MODELS.md`). No schema migration is required:
the column is `TEXT` and existing rows are unaffected; only the
validator threshold changes.

## Known issues

The two SPEC-vs-code divergences flagged in v1.3.0 remain open and are
unchanged by this release. Both are tracked as follow-up issues for a
future `specification-manager` reconciliation pass; neither affects
runtime behaviour.

1. **Exit-code mapping for `ErrInvalidInput`.**
   `SPEC/ARCHITECTURE.md` § *Error Reuse Policy / Sentinel Error
   Catalogue* states that `ErrInvalidInput` maps to exit code `2`. The
   implementation in `cmd/rmp/main.go` (`handleError`) maps
   `ErrInvalidInput`, `ErrValidation` and `ErrFieldTooLarge` to
   `ExitInvalidData = 6`, and the help texts shipped in this release
   continue to reflect the implementation.

2. **`audit stats` JSON keys.**
   `SPEC/COMMANDS.md` § *Audit Statistics* documents the keys
   `operations_count`, `entity_type_count`, `first_entry`, `last_entry`
   and a `period.{since,until}` block. The implementation
   (`internal/models/audit.go` and `internal/commands/audit.go`) emits
   `by_operation`, `by_entity_type`, `first_entry_at`, `last_entry_at`
   and `total_entries`, with no `period` object.

## Upgrade notes

No action required. v1.4.0 is fully backward compatible with v1.3.0.
Existing roadmap databases, JSON outputs, exit codes and command syntax
all continue to work unchanged. AI-agent consumers that wish to adopt the
contract can do so opportunistically:

- detect support by invoking `rmp --ai-help`; on older versions this
  exits with `127` (unknown command) and prints standard help on
  stderr;
- the contract JSON is self-describing and includes a top-level
  version field that agents can pin against.

To upgrade:

```sh
git fetch --tags
git checkout v1.4.0
make check
go build -o ./bin/ ./cmd/rmp
```

Or, for users who installed via `install.sh`, simply re-run the
installer against the new tag.
