## Executive Summary

Groadmap v1.11.0 is a reliability release for the read-only `rmp web` server of
the `rmp` roadmap-management CLI. It makes `rmp web` **auto-migrate every served
roadmap's SQLite schema once at startup**, before the HTTP listener binds. This
closes a gap in which `rmp web`, when it was the first command run against a
roadmap whose on-disk schema predated a binary schema bump, returned HTTP 500 on
the sprints page because the read-only query referenced columns that the stale
database file did not yet contain.

The migration runs through the writable `db.Open` path, whose idempotent
`RunMigrations` is a no-op when the schema is already current, and then closes the
database. The per-request data path is untouched and remains **strictly
read-only** (`OpenReadOnly` with SQLite `query_only`), so no write, audit entry,
or schema change can ever occur while serving a request. This preserves the
read-only invariant established as audit finding #43.

Under Semantic Versioning 2.0.0 this is a **MINOR** release: the change is
additive and fully backward compatible with `v1.10.0`. No `rmp` command, flag,
JSON output, exit code, or on-disk format is altered. The database schema version
is unchanged at `1.8.0`; this release introduces no migration of its own and only
applies the already-defined migrations earlier in the `rmp web` lifecycle.

## Changes by Category

### Added â€” Startup Schema Migration for `rmp web`

The `serve()` lifecycle gains a new `migrateRoadmapsAtStartup()` step, invoked
after `EnsureDataDir` and before the listener binds. It enumerates every roadmap
via `utils.ListRoadmaps()`, opens each one through the writable `db.Open` path â€”
which runs the idempotent `RunMigrations`, a no-op when the schema is already
current â€” and then closes it. Because this step runs before any read-only
connection is opened, every roadmap the server will serve is guaranteed to be at
the current schema by the time the first request arrives.

The step is deliberately resilient. A failure to list, open, or migrate any
single roadmap is written to stderr and is **non-fatal**: the server continues
and still serves the roadmaps that did migrate successfully, so one damaged or
inaccessible roadmap cannot prevent the whole web interface from starting.

This behaviour is specified in `SPEC/WEB.md` (Â§ Startup Schema Migration, Server
Lifecycle step 2, Acceptance Criteria 41/42 and the Read-Only Data Flow
clarification) and `SPEC/ARCHITECTURE.md` (Â§ internal/web, refined to note the
single startup write).

### Fixed â€” `rmp web` Sprints Page Returned HTTP 500 on a Stale Schema

Before this release, when `rmp web` was the first command run after a binary
schema bump on a roadmap whose `project.db` predated schema `1.7.0`/`1.8.0`, the
sprints page (`GET /roadmaps/{name}`) failed with HTTP 500. The read-only query
referenced columns absent from the stale file â€” `sprints.title` (schema `1.7.0`)
and `sprints.order_index` (schema `1.8.0`).

The root cause is structural: the per-request loaders open the database read-only
(`OpenReadOnly`, `query_only`) and therefore cannot migrate it, and `rmp web` may
be the very first command a user runs against a roadmap after upgrading the
binary. Migrating once at startup, before any read-only connection is opened,
closes this gap. The fix makes migration automatic and input-free while keeping
every per-request connection strictly read-only: no write, no audit row, and no
schema change occurs on a read (finding #43 preserved).

## Breaking Changes

None. The public CLI command surface, every flag, all JSON output schemas, exit
codes, the on-disk graph format, and the database schema are fully backward
compatible with `v1.10.0`. The only behavioural change is that `rmp web` now
brings a stale served roadmap to the current schema at startup instead of failing
to render its sprints page.

## Upgrade Notes

No manual migration steps are required. This release adds no SQLite migration of
its own; the database schema version is unchanged at `1.8.0`. If a served roadmap
is at an older schema (for example after upgrading the binary), `rmp web` now
migrates it automatically on startup. The migration is idempotent, so starting
`rmp web` against already-current roadmaps does no work and changes nothing.

```sh
# Build from source
git fetch --tags
git checkout v1.11.0
go build -o ./bin/rmp ./cmd/rmp

# Verify
./bin/rmp --version
# Groadmap version 1.11.0
```

## Benchmark Evidence

This release adds a single startup step to `rmp web` and changes no CLI data path,
so the Groadmap CLI command benchmarks are byte-for-byte the same code as
`v1.10.0` and are not re-measured here; the baselines established in `v1.3.0`
remain the reference. The new startup migration runs once per server start, before
the listener binds, and is a no-op when every served roadmap is already at the
current schema, so it adds no per-request cost. Its correctness is governed by the
idempotency and read-only contracts validated functionally by the unit and E2E
suites above rather than by a throughput target.
