## Executive Summary

Groadmap v1.7.0 introduces the **`rmp graph`** command family: a per-roadmap
knowledge graph that lets an AI agent record and query the project's elements
and the relationships between them â€” a queryable "second brain" for a roadmap.
The graph is accessed through Cypher and exposed via five guard-railed
subcommands (`create`, `query`, `update`, `delete`, `search`). It is backed by
the external GoGraph engine, which supplies a labelled property graph, a Cypher
engine, and a durable on-disk store. Every write is made durable through a
synchronous checkpoint that snapshots the committed state and truncates the
write-ahead log before the process exits.

The graph is a purely additive capability. The existing CLI surface, all exit
codes, and every JSON output schema remain fully backward compatible with
`v1.6.0`, so this is a **MINOR** release under Semantic Versioning. Existing
`v1.6.0` installations require no migration.

## Changes by Category

### Features â€” The `rmp graph` Command Family

The knowledge graph turns a roadmap into a queryable knowledge space. Groadmap
does not impose a fixed schema on it: the agent decides which nodes, edges,
labels, and properties to model. Each roadmap owns exactly one graph. In this
first version the graph is independent of the roadmap's tasks and sprints data;
graph operations never read or write `project.db`.

Every subcommand requires a target roadmap via the shared `-r` / `--roadmap`
flag, and reads its Cypher from the `--query` flag when present, otherwise from
standard input. This precedence enables both inline invocation and piped or
heredoc usage.

| Subcommand | Operation class | Behaviour |
|------------|-----------------|-----------|
| `graph create` | Write | Execute write Cypher that adds nodes and edges. |
| `graph query` | Read | Execute read Cypher; return `columns`/`rows` as JSON. |
| `graph update` | Write | Execute write Cypher that modifies existing elements. |
| `graph delete` | Write | Execute write Cypher that removes elements (durable tombstones). |
| `graph search` | Read | Execute read Cypher tailored to lookup and traversal queries. |

**Guard-rail validation.** Each subcommand validates that the supplied Cypher
matches its operation class (read vs. write) before executing it. A read
subcommand therefore cannot mutate the graph, and a write subcommand cannot be
repurposed to bypass the intended access pattern.

**Result shapes.** Read subcommands (`query`, `search`) return their result
columns and rows as JSON on stdout. Write subcommands (`create`, `update`,
`delete`) run inside a single transaction and mirror the query's `RETURN`
clause: a query with `RETURN` returns the same `columns`/`rows` shape, while a
query without `RETURN` returns `{"ok": true}`.

### Persistence â€” Synchronous Checkpoint on Write

After a write subcommand commits its transaction durably, and before the
process exits, the implementation produces a self-sufficient on-disk snapshot
of the committed graph state and truncates the write-ahead log â€” synchronously,
within the same invocation. This checkpoint bounds write-ahead-log growth and
keeps recovery cost proportional to the live graph size rather than to the
total history of writes. Read subcommands (`query`, `search`) are read-only and
never checkpoint. The contract is specified in
`SPEC/GRAPH.md Â§ Synchronous Checkpoint on Write`.

The graph is persisted per roadmap under `~/.roadmaps/<name>/graph/`. Deletions
are recorded as durable tombstones that survive a store reopen, and the store
supports multigraph parallel edges, allowing multiple distinct relationships
between the same pair of nodes.

## Upgrade Notes

No migration steps are required. The graph store is a separate GoGraph store
under `~/.roadmaps/<name>/graph/`; it is created lazily on first graph use and
does not touch the roadmap's SQLite `project.db`. The database schema version is
unchanged at `1.6.0` â€” the graph feature adds no SQLite migration.

```sh
# Build from source
git fetch --tags
git checkout v1.7.0
go build -o ./bin/rmp ./cmd/rmp

# Verify
./bin/rmp --version
# Groadmap version 1.7.0
```
