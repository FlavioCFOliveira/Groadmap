# Release Notes — Groadmap v1.10.0

**Date:** 2026-06-16
**Type:** Minor (features, reliability, and security)
**Tag:** `v1.10.0`
**Previous release:** `v1.9.1` (2026-06-05)

---

## Executive Summary

Groadmap v1.10.0 is a feature and reliability release for the `rmp` roadmap-
management CLI. It enriches sprints with a required, human-readable **title** and
a unique **execution-order** field; sharpens the read-only `rmp web` interface so
it never serves stale data, presents every sprint through one shared template,
preserves authored free-text line breaks, and runs a hardened HTTP server; and
surfaces GoGraph query notifications as stderr diagnostics. Underneath the new
functionality, the release lands two complete audits — a 32-finding reliability
and specification-conformance audit (findings #39–#63) and a 23-finding security
audit (findings #64–#87) — each finding closed with a permanent regression test.
The knowledge-graph engine **GoGraph** is upgraded `v0.2.0` → `v0.3.2`.

Under Semantic Versioning 2.0.0 this is a **MINOR** release. Every change is
additive or corrective and remains backward compatible with `v1.9.1`, with one
deliberate command-line exception: `sprint create` now requires `--title`. The
database schema advances to `1.8.0` through two automatic, idempotent migrations;
existing installations upgrade transparently on first run.

---

## Highlights

- **Sprint title:** sprints now carry a required, human-readable title (max 255
  characters) alongside their description, so each sprint is identifiable at a
  glance. `sprint show` output gains a `sprint_title` field.
- **Sprint execution order:** every sprint carries a unique, positive `--order`,
  auto-assigned when omitted and editable while the sprint is open. Listings sort
  by it deterministically.
- **`rmp web` never serves stale data:** `Cache-Control: no-store` on every data-
  derived response guarantees each reload reflects the exact current
  database/graph state.
- **Unified sprint presentation:** the sprint page and the sprints "Actual" tab
  render the same shared sub-template, and the web summary shares its arithmetic
  with the CLI `sprint show` report, so the two can never diverge.
- **Hardened web server:** request timeouts, a strict security-header set,
  loopback-by-default binding, no directory listing, and HTML-safe graph JSON.
- **Graph query notifications:** `rmp graph query`/`search` surface GoGraph
  advisories (for example the Cartesian-product warning) on stderr without
  altering stdout JSON or exit codes.
- **Two audits closed:** 32 reliability/spec-conformance findings and 23 security
  findings resolved, each guarded by a regression test.
- **GoGraph `v0.2.0` → `v0.3.2`:** robustness, security, Cypher, and durability
  hardening; fixes the v0.3.0/v0.3.1 recovery panic on `v0.2.0`-written stores.

## Changes by Category

### Added — Required Sprint Title

Sprints gain a required, human-readable title in addition to their description.
`sprint create` requires `-t/--title` (maximum 255 characters); `sprint update`
accepts an optional `-t/--title`. Both the registry-driven validation/`--ai-help`
contract and the hand-written static help screens advertise the flag, so the two
cannot drift apart.

The schema adds a `title TEXT NOT NULL CHECK(length(title) <= 255)` column. The
migration to schema `1.7.0` adds the column with a temporary `DEFAULT ''` (so
SQLite can add a `NOT NULL` column to a populated table) and backfills each
existing sprint with `Sprint <id>`; it is idempotent (column-existence guard plus
a `WHERE title = ''` clause so a re-run never clobbers a real title). The
`sprint show` JSON output gains a `sprint_title` field.

Specified across `SPEC/MODELS.md`, `SPEC/DATABASE.md`, `SPEC/COMMANDS.md`,
`SPEC/DATA_FORMATS.md`, and `SPEC/VERSION.md`.

### Added — Unique Sprint Execution-Order Field

Every sprint carries a unique, positive execution order. `sprint create` accepts
an optional `--order`; when omitted it is auto-assigned `MAX(order_index) + 1`
atomically inside the create transaction. `sprint update` may edit `--order`
while the sprint is `PENDING` or `OPEN` and rejects an edit on a `CLOSED` sprint
with exit 6 (reopening restores mutability). A non-positive or non-integer value
is exit 6; a collision with an existing order maps to exit 5.

The schema adds `order_index INTEGER NOT NULL CHECK(order_index > 0)` and a unique
index. The migration to schema `1.8.0` adds the column with a temporary
`DEFAULT 0`, deterministically backfills `1..N` by `(created_at ASC, id ASC)`,
then creates the unique index; it is idempotent via a column-existence guard.
`sprint get` and `sprint list` expose the `order` field, and the web sprints page
orders upcoming sprints ascending and concluded sprints descending by it.

Specified across `SPEC/MODELS.md`, `SPEC/DATABASE.md`, `SPEC/STATE_MACHINE.md`,
`SPEC/COMMANDS.md`, `SPEC/HELP.md`, `SPEC/DATA_FORMATS.md`, `SPEC/WEB.md`, and
`SPEC/VERSION.md`.

### Added — Graph Query Notifications as Diagnostics

`rmp graph query` and `rmp graph search` now print each GoGraph query
notification as a plain-text line to stderr in the form
`<Severity> <Code>: <Description>`. Today the engine emits the Neo4j-compatible
`CartesianProductWarning` for a disconnected multi-pattern `MATCH`. The wiring is
present on both the read and write paths; with the pinned GoGraph v0.3.2 the
engine attaches notifications to the read-path result only, so write subcommands
emit none yet, and the surfacing activates automatically if a future engine
carries them on the transactional path. The stdout success JSON and exit codes
are unchanged, consistent with the existing checkpoint stderr-diagnostic pattern.
Specified in `SPEC/GRAPH.md`.

### Changed — `sprint create` Now Requires `--title`

`sprint create` now treats `-t/--title` as a required parameter. A call without
it fails with exit 6 and the SPEC-canonical message
`required parameter missing: --title`. This is the only backward-incompatible
command-line change in the release; see Upgrade Notes for the one-line script
adjustment.

### Changed — Unified Sprint Presentation in `rmp web`

A shared `{{sprintDetail}}` sub-template renders the full sprint block — the
completion-summary line, the metadata datagrid (ID, status, capacity, tasks,
created/started/closed), and the member-task table — identically on the sprint
page and the "Actual" tab of the sprints page. The CLI `sprint show` report and
the web summary now share `CategorizeTaskStatus`, `CalculateSprintSummary`, and
`CompletionPercentage`, so their figures are guaranteed consistent. The summary
line format is `<pct>% - P:<p> A:<a> C:<c> - T:<t>`. Specified in `SPEC/WEB.md`.

### Changed — Preserve Authored Line Breaks in the Graph Detail Panel

The `rmp web` knowledge-graph detail panel now tags each property value with a
dedicated class and renders it with `white-space: pre-wrap`, so node and edge
properties authored as multi-line free text (for example a node's specification
or notes) preserve their source line breaks while still wrapping. Values are
assigned via `textContent` and are never interpreted as HTML. Specified in
`SPEC/WEB.md` (Frontend Rules rule 6).

### Changed — GoGraph Upgraded `v0.2.0` → `v0.3.2`

The engine behind `rmp graph` and the read-only graph page of `rmp web` is
upgraded to GoGraph `v0.3.2`, adopting upstream robustness and security hardening
(three audits, including SEC-2026-06-14b/c), Cypher-engine and
WAL/recovery/snapshot durability and correctness fixes, and a benchstat-gated
performance cycle. The release line is API-additive over `v0.2.0`: no exported
identifier consumed by `internal/commands/graph.go` or `internal/web/data.go`
across `cypher`, `cypher/expr`, `graph/csr`, `graph/lpg`, and
`store/{recovery,snapshot,txn,wal}` was removed or renamed. The pin targets
v0.3.2 specifically because v0.3.0/v0.3.1 carried a recovery panic
(`makeslice: cap out of range`) when reopening a `v0.2.0`-written store, which
v0.3.2 fixes. The indirect `golang.org/x/sys` hash is refreshed and the toolchain
directive is `go1.26.4`. Compatibility was verified empirically: the existing
project knowledge graph opens and read/writes correctly under v0.3.2. Specified
across `SPEC/BUILD.md`, `SPEC/GRAPH.md`, and `SPEC/ARCHITECTURE.md`.

### Fixed — Reliability and Specification Conformance (findings #39–#63)

- **Concurrent graph-write data loss (#39, CRITICAL):** `rmp graph` writes take
  an exclusive, non-blocking file lock for the entire write (open → commit →
  checkpoint → WAL-truncate). Two concurrent writers could otherwise interleave
  so one writer's full-snapshot checkpoint overwrote the other's committed-but-
  unseen write and then truncated the WAL that still held it, silently losing an
  acknowledged write. Contention now surfaces cleanly as exit 1.
- **Per-connection PRAGMAs (#41):** `foreign_keys` and `busy_timeout` are carried
  in the SQLite DSN so every pooled connection applies them. A one-shot `Exec`
  had configured only the first connection, leaving the second pooled connection
  with `foreign_keys=OFF` (silently disabling `ON DELETE CASCADE`) and
  `busy_timeout=0`.
- **Numeric migration comparison (#42):** migration gating compares versions
  component-wise and numerically, so `1.9.0` versus `1.10.0` orders correctly; a
  lexicographic compare would have skipped migrations once a component reached two
  digits.
- **Read-only web database access (#43):** `rmp web` opens roadmap databases with
  `query_only(true)` and without running migrations, so a page view can never
  rewrite a stale-schema `project.db`.
- **Task-command correctness (#44–#46, #48):** `task get`, `task priority`, and
  `task severity` fail fast with exit 4 on unknown IDs, writing no phantom audit
  rows and mutating no valid task in a mixed batch; out-of-range priority/severity
  returns exit 6; a no-field `task edit` is a successful no-op.
- **Sprint task management (#40, #47, #49, #50):** `sprint remove-tasks` is scoped
  to the named sprint (membership-checked, exit 6 otherwise), resets reverted
  tasks to `BACKLOG` clearing their lifecycle timestamps and completion summary,
  and compacts remaining positions to a contiguous `0..N-1` sequence; `move-to`
  clamps an out-of-range position to the end.
- **State-machine and empty-list contracts (#53, #55):** the manual
  `DOING → SPRINT` transition is forbidden (SPRINT is set exclusively by
  `sprint add-tasks`); empty sprint and audit lists marshal to `[]`, not `null`.
- **SPEC-verbatim messages and exit codes (#54, #58, #59, #60):** required-
  parameter and roadmap-name messages match the SPEC canonical text exactly; an
  int64-overflowing all-digit ID is a range failure (exit 6); the data directory's
  `0700` permission is re-applied and verified on every layout migration.
- **Global help generated from the registry (#51):** `rmp --help` builds its
  command list from the single command registry, which had silently dropped the
  `web` command from the hardcoded block.
- **Atomic audit and transactional integrity (#65–#67):** `AddTasksToSprint` and
  `MoveTasksBetweenSprints` write their audit rows inside the same transaction as
  the membership change; sprint capacity is enforced inside the transaction,
  closing a TOCTOU window; `DeleteSprint` and `RemoveTasksFromSprint` run their
  multi-statement mutations in a single transaction.
- **Idempotent migrations (#68):** `ALTER TABLE ADD COLUMN` migrations are guarded
  by a `pragma_table_info` column-existence check.
- **Audit catalogue cleanup:** the unused `TASK_TYPE_CHANGE` audit operation is
  removed (a type change is recorded as `TASK_UPDATE`); help, SPEC, and code agree.

### Security — Hardening (findings #64–#87)

| Area | Findings | Hardening |
|------|----------|-----------|
| `rmp web` DoS | #69 | `WriteTimeout` 30 s and `IdleTimeout` 120 s added |
| Directory disclosure | #70 | `/static/` directory requests return 404 |
| Security headers | #71 | CSP, `X-Content-Type-Options`, `X-Frame-Options`, `Referrer-Policy` on every response |
| JSON injection | #73 | `/graph/data` emits HTML-safe JSON |
| Network exposure | #76 | default bind `127.0.0.1`; warning on non-loopback bind |
| Stale data | — | `Cache-Control: no-store` on every data-derived response |
| Symlink following | #72, #74, #75 | `Lstat` guard on data dir, roadmap dirs, and legacy migration |
| Unbounded scan | #64 | `GetAuditEntries` hard-caps its result set |
| File permissions | #77, #78 | `project.db` created `O_CREATE\|O_EXCL` `0600`; `-wal`/`-shm` `chmod` `0600` |
| Input validation | #82–#87 | control-char/bidi rejection (CVE-2021-42574), list-separator and numeric bounds |
| Graph guard-rail | #79, #80 | read subcommands reject `CREATE`/`DROP INDEX\|CONSTRAINT` (literal-aware) |

The input-validation hardening rejects ASCII control characters (except
TAB/LF/CR), DEL, and Unicode bidirectional/format code points across CLI free-text
inputs, blocking terminal-escape injection and Trojan Source (CVE-2021-42574),
while leaving legitimate Unicode unaffected. The graph guard-rail uses a case- and
whitespace-insensitive matcher on the literal-masked query, so DDL keywords inside
string literals — for instance text stored in the knowledge-graph memory — are not
misclassified.

## Breaking Changes

One command-line behaviour changed: **`sprint create` now requires `-t/--title`**.
A call to `sprint create` without `--title` fails with exit 6
(`required parameter missing: --title`) rather than creating a sprint. No other
command, flag, JSON output schema, exit code, or on-disk format is broken. The
two schema migrations (`1.7.0`, `1.8.0`) are automatic and idempotent, and they
backfill every existing sprint, so they neither lose data nor require manual
intervention.

## Upgrade Notes

The two schema migrations run automatically on the first command against an
existing roadmap. Each is idempotent and backfills existing rows (titles as
`Sprint <id>`, execution order as `1..N` by creation order), so no manual step is
required.

Update any automation that creates sprints to pass a title:

```sh
# Before (v1.9.1 and earlier)
rmp sprint create -r myroadmap --goal "Stabilise the storage layer" \
  --start 2026-07-01 --end 2026-07-14

# After (v1.10.0)
rmp sprint create -r myroadmap --title "Storage hardening" \
  --goal "Stabilise the storage layer" --start 2026-07-01 --end 2026-07-14
```

Build and verify the release:

```sh
git fetch --tags
git checkout v1.10.0
go build -o ./bin/rmp ./cmd/rmp

./bin/rmp --version
# Groadmap version 1.10.0
```

If you operate `rmp graph` on a store damaged by a hardware fault or an out-of-
band edit, note that the GoGraph v0.3.x recovery path fail-stops on genuine
corruption; this is intentional and prevents new writes from being appended onto
a damaged store.

## Benchmark Evidence

This release changes correctness, validation, and presentation paths rather than
the throughput-critical data paths, and it is governed by correctness and crash-
safety contracts validated functionally by the unit and E2E suites above. The CLI
data-path benchmark baselines established in v1.3.0 remain the reference and are
not re-measured here. On the dependency side, GoGraph v0.3.2 ships a benchstat-
gated performance cycle upstream; the algorithms it tunes are not on a Groadmap
hot code path, and Groadmap's consumed surface is unchanged at the API level.

## Known Issues

One SPEC-vs-code divergence remains open and is unaffected by this release:

1. **`audit stats` JSON keys** — `SPEC/COMMANDS.md` documents the keys
   `operations_count`, `entity_type_count`, `first_entry`, `last_entry` and a
   `period.{since,until}` block; the implementation emits a different (stable) set
   (`by_operation`, `by_entity_type`, `first_entry_at`, `last_entry_at`,
   `total_entries`, with no `period` object).

This divergence does not affect runtime behaviour. It is tracked as a `spec` /
`tech-debt` GitHub issue and will be resolved by a `specification-manager` pass in
a future release.
