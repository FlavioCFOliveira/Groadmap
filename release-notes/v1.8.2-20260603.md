## Executive Summary

Groadmap v1.8.2 corrects a silent data-truncation defect in large multi-task
operations and activates a query-caching optimisation that was specified but
dormant. The headline fix removes a hidden 1000-id cap: `task list` and the batch
update paths now return and update the complete set of matching tasks instead of
quietly stopping at the first 1000. The companion fix reconciles the query cache
with the real database schema so the optimisation mandated by
`SPEC/IMPLEMENTATION.md` (Â§ Query Caching) finally runs on repeated batch
operations.

Both changes are backward-compatible bug fixes. There are no new commands, flags,
JSON output fields, exit codes, or database schema changes, and the AI contract
surface (`rmp --ai-help`) is unchanged. Under Semantic Versioning 2.0.0 this is a
**PATCH** release, fully compatible with `v1.8.1`. The database schema version is
unchanged at `1.6.0`, so existing installations require no migration.

## Changes by Category

### Fixed â€” Large Task Fetches and Batch Updates No Longer Truncate to 1000 Ids

**Symptom.** On a large roadmap, `task list` and other multi-id task operations
could silently return or update only the first 1000 matching tasks. Tasks beyond
that cap were missing from results, and batch status, priority, and
sprint-membership updates could skip them without any error.

**Root cause.** Each multi-id path built a single `WHERE id IN (...)` clause and
passed the id count through `normalizeSize`, whose upper bound is 1000. The id
set was therefore clamped to 1000 before the query ran, so any larger set was
truncated. The behaviour was silent: no error, no warning, just an incomplete
result.

**Fix.** Every multi-id path now sorts a copy of the id set and chunks it through
the `BatchProcessor` (`ProcessChunks` for write paths and
`ProcessChunksWithResult` for `GetTasks`). Each chunk fetches a cached template
sized to the chunk and runs within SQLite's per-statement variable limit, and the
per-chunk results are concatenated in deterministic id order. The caller's slice
is never mutated (a sorted copy is used), so callers see no side effects beyond
the now-complete result set.

**Impact.** Commands that operate over many tasks now return and modify the full
set. Small roadmaps (1000 tasks or fewer) are unaffected in behaviour; they
simply run through a single chunk.

### Fixed â€” Query Cache Reconciled With the Real Schema and Activated

**Symptom.** The query cache, mandated by `SPEC/IMPLEMENTATION.md` (Â§ Query
Caching), did not actually accelerate anything in production. Despite the cache
existing in the code, batch operations rebuilt their SQL on every call.

**Root cause.** The cached templates referenced a fictional schema â€” columns that
never existed in the `tasks` table. The templates could not match the real
queries, so the cached plans were never used and the optimisation was effectively
dead code.

**Fix.** Templates are now generated from a single `buildTemplates` function that
is the shared source of truth for both the pre-generation path
(`initializeTemplates`) and the on-demand fallback. Each template is
byte-identical in semantics to the corresponding production query: the full task
projection with its dependency columns, the correlated subtask-count subquery,
and the `ORDER BY t.id` tail for reads; the appropriate timestamp column per
status transition for writes. With the templates matching reality, the cache and
batch path are now genuinely active.

### Performance â€” Repeated Batch Operations Reuse Cached Query Plans

With the query cache reconciled and activated, `GetTasks` and the batch
status/priority/sprint-membership update paths now fetch a cached, chunk-sized
template per operation instead of formatting a fresh SQL string on every call.
The optimisation is internal and changes no observable output. It reduces
per-call query construction on hot batch paths; the established v1.3.0 benchmark
baselines remain the reference for these paths (see Benchmark Evidence).

### Removed â€” Dead Code in `internal/db` and `internal/commands`

Ten unused `internal/db` functions were removed: the parent/subtask helpers
(`GetParentTask`, `GetIncompleteSubTasks`, `HasSubTasks`), the task-dependency
helpers (`AddTaskDependency`, `RemoveTaskDependency`,
`GetIncompleteDependencies`, `GetTaskDependsOn`, `GetTaskBlocks`), and the
max-position helpers (`getMaxPositionInternal`, `GetMaxPosition`). The unused
`HandleBacklog` and `HandleGraph` command wrappers were also removed, since
command dispatch already routes through the central registry. All removed paths
were unreachable; there is no behaviour change.

### Tests â€” Dormant Suites Revived and a Dormancy Guard Added

Nine `tests/test_*.py` files existed on disk but were never registered in
`tests/run_tests.py`, so they never ran and produced a false sense of coverage.
All nine are now registered, and a new `assert_no_dormant_modules()` guard fails
the run immediately if any `tests/test_*.py` file is left unregistered, so the
gap cannot silently return. The registered E2E suite grows from 27 to 37 modules.
A new suite, `tests/test_38_task_list_date_filters.py`, covers the
`task list --created-since` / `--until` date filters. Two stale tests targeting
non-existent features were removed.

New Go unit tests accompany the fixes: `internal/db/batch_test.go` and
`internal/db/query_cache_test.go` for the chunking and cache reconciliation, and
`internal/web/data_test.go`, `internal/web/handlers_test.go`, and
`internal/web/server_test.go` for the web server.

### Tooling â€” New Coverage Targets

`make cover` reports unit-test coverage. `make cover-full` builds an instrumented
binary, drives it through the full E2E suite, and merges the result with unit
coverage to report the real exercised command surface. The web server now tears
down gracefully on `SIGTERM`, so coverage counters flush on shutdown and web
coverage becomes measurable.

## Benchmark Evidence

No dedicated benchmark was re-run for this release. The performance change is the
activation of an already-specified query cache and the chunking of multi-id paths
through the `BatchProcessor`; it alters query construction, not algorithmic
complexity, and produces byte-identical results. The benchmark baselines
established in v1.3.0 remain the reference for the affected batch and fetch paths.
The repository's `internal/db/bench_test.go` benchmarks were updated alongside the
chunking change and remain available for ad-hoc measurement via
`go test -bench=. ./internal/db`.

The correctness fix is the primary user-visible effect: on roadmaps with more
than 1000 tasks, operations that previously returned 1000 rows now return the
complete set, which is the intended behaviour rather than a performance
regression.
