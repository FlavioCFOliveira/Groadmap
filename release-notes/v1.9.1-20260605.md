# Release Notes — Groadmap v1.9.1

**Date:** 2026-06-05
**Type:** Patch (dependency hardening)
**Tag:** `v1.9.1`
**Previous release:** `v1.9.0` (2026-06-03)

---

## Executive Summary

Groadmap v1.9.1 hardens the `rmp graph` knowledge-graph store by upgrading its
backing engine, **GoGraph**, from `v0.1.0` to `v0.2.0`. GoGraph v0.2.0 is a
reliability, ACID, and durability hardening release; the parts of it that Groadmap
consumes gain stronger crash-safety and a fail-stop response to genuine store
corruption, and the upgrade also pulls in the `go1.26.4` toolchain, which resolves
two reachable Go standard-library vulnerabilities.

This is a **drop-in** dependency upgrade. No `rmp` command, flag, JSON output,
exit code, on-disk graph format, or database schema is altered, and no Groadmap
source change is required. A v0.2.0 usage evaluation confirmed that the existing
consumers — `internal/commands/graph.go` and `internal/web/data.go` — are
source-compatible against the new API, and the only behavioural change that
reaches Groadmap strengthens existing error handling on the recovery path. Under
Semantic Versioning 2.0.0 this is a **PATCH** release, fully backward compatible
with `v1.9.0`. The database schema version is unchanged at `1.6.0`, so existing
installations require no migration.

## Highlights

- **Hardened graph store:** the GoGraph engine behind `rmp graph` (and the
  read-only graph page of `rmp web`) is upgraded `v0.1.0` → `v0.2.0`, a
  reliability, ACID, and durability hardening release.
- **Fail-stop on genuine corruption:** the graph store now returns a clean error
  on genuine write-ahead-log corruption (CRC mismatch or unsupported record
  version) instead of silently swallowing it, while a benign crash-truncated tail
  still recovers cleanly.
- **Inherited crash-durability fixes:** snapshot staging-directory `fsync` before
  the publish rename, durable-before-visible autocommit writes, and a persisted
  directed/multigraph shape in the snapshot manifest.
- **Two security advisories resolved:** the `go1.26.4` toolchain that GoGraph
  v0.2.0 requires resolves **GO-2026-5039** (`net/textproto`) and
  **GO-2026-5037** (`crypto/x509`); `govulncheck ./...` reports no vulnerabilities.
- **Drop-in:** no CLI, flag, JSON output, exit-code, on-disk graph format, or
  database schema change. Fully backward compatible with `v1.9.0`; schema version
  unchanged at `1.6.0`.

## Changes by Category

### Changed — `rmp graph` Store Hardened via GoGraph `v0.1.0` → `v0.2.0`

The knowledge-graph store that backs the `graph` command family — a GoGraph store
rooted at `~/.roadmaps/<name>/graph/`, separate from `project.db` — is upgraded to
GoGraph `v0.2.0`. GoGraph v0.2.0 is a pre-1.0 **MINOR** release whose headline is
a deep reliability, ACID, and durability hardening pass across the persistence
stack, the Cypher engine, and the Bolt server. It is API-additive over `v0.1.0`:
no exported identifier that Groadmap consumes was removed or renamed.

The exact-tag pin in `go.mod` moves from `v0.1.0` to `v0.2.0`, and the indirect
`golang.org/x/exp` hash is refreshed. The consumed surface is unchanged at the API
level: `store/recovery`, `store/wal`, `store/txn`, `store/snapshot`, `cypher`,
`cypher/expr`, `graph/lpg`, and `graph/csr`. A v0.2.0 usage evaluation confirmed
there are no required code changes in Groadmap; the existing consumers
(`internal/commands/graph.go` and `internal/web/data.go`) compile and behave
unchanged on the normal path. The specification was reconciled to the new pin
across `SPEC/BUILD.md`, `SPEC/GRAPH.md`, and `SPEC/ARCHITECTURE.md`.

### Fixed — Fail-Stop on Genuine Graph-Store Corruption

`recovery.Open` is the function Groadmap uses to open the GoGraph store read-only
for every `rmp graph` query/search and for the read-only graph page of `rmp web`.
Under GoGraph v0.2.0, recovery now **fail-stops on genuine corruption**: a CRC
mismatch or an unsupported record version is returned to the caller as a clean
error, instead of the v0.1.0 behaviour of swallowing it and risking further
appends onto a damaged store. A benign torn or crash-truncated WAL tail remains a
clean success, and GoGraph reports the distinction via `Result.IsClean`, so the
change tightens only the genuine-corruption path and leaves the normal open path
untouched.

The upgrade also inherits GoGraph's crash-durability ordering fixes, which apply
to the store Groadmap writes on every `rmp graph` mutation:

- The snapshot writer `fsync`s its staging directory before the publish rename
  (write-and-`fsync` each component → `fsync` the directory → rename → `fsync` the
  parent).
- Cypher autocommit mutations are made durable (WAL `fsync`) before they become
  visible to concurrent readers, so a reader can never observe a write that a
  crash would then lose.
- The snapshot manifest now records the graph's directed/multigraph shape, so a
  graph created simple no longer silently becomes a multigraph after a reopen;
  manifests without the field default to the previous behaviour, so existing
  snapshots replay unchanged.

### Security — Two Go Standard-Library Vulnerabilities Resolved

GoGraph v0.2.0 requires the **`go1.26.4`** toolchain, which resolves two Go
standard-library vulnerabilities reachable through the dependency:

| Advisory | Package | Reached via |
|----------|---------|-------------|
| **GO-2026-5039** | `net/textproto` | snapshot open path (`textproto.Reader.ReadMIMEHeader`) |
| **GO-2026-5037** | `crypto/x509` | the GoGraph Bolt TLS handshake |

`govulncheck ./...` was run against the upgraded module this release cycle and
reports **"No vulnerabilities found."** The Go *language* directive is unchanged;
only the toolchain directive moved, so no new language features are adopted.

### Documentation — Regression Prevention Principle

`CLAUDE.md` gains a "Regression Prevention" working principle. This is an
internal, contributor-facing governance change only. It has no code, CLI, JSON,
exit-code, or runtime impact and is not user-visible.

## Breaking Changes

None. The public CLI command surface, every flag, all JSON output schemas, exit
codes, the on-disk graph format, and the database schema are fully backward
compatible with `v1.9.0`. The GoGraph upgrade is a drop-in: the only behavioural
change that reaches Groadmap is the stricter, safer corruption response on the
graph-store recovery path, which can only convert a previously-silent corrupted
open into a clean, reported error.

## Upgrade Notes

No migration steps are required. The graph store adds no SQLite migration; the
database schema version is unchanged at `1.6.0`. Existing graph stores at
`~/.roadmaps/<name>/graph/` are read and written unchanged.

```sh
# Build from source
git fetch --tags
git checkout v1.9.1
go build -o ./bin/rmp ./cmd/rmp

# Verify
./bin/rmp --version
# Groadmap version 1.9.1
```

If you operate `rmp graph` on a store that was damaged by a hardware fault or an
out-of-band edit, be aware that an open which previously appeared to succeed
silently may now fail fast with a clean corruption error. This is the intended,
safer behaviour: it prevents new writes from being appended onto a damaged store.

## Benchmark Evidence

This release introduces no Groadmap source change, so the Groadmap CLI data paths
are byte-for-byte the same code as `v1.9.0`; the benchmark baselines established
in v1.3.0 remain the reference and are not re-measured here. On the dependency
side, GoGraph v0.2.0 reports its single-threaded benchmark figures as unchanged
from v0.1.0, with one memory-footprint improvement to Personalised PageRank whose
rank vector is bit-for-bit identical; that algorithm is not on a Groadmap code
path. The hardening work is governed by correctness and crash-safety contracts
rather than by a throughput target and is validated functionally by the unit and
E2E suites above.

## Known Issues

One SPEC-vs-code divergence remains open and is unaffected by this release:

1. **`audit stats` JSON keys** — `SPEC/COMMANDS.md` documents the keys
   `operations_count`, `entity_type_count`, `first_entry`, `last_entry` and a
   `period.{since,until}` block; the implementation emits a different (stable) set
   (`by_operation`, `by_entity_type`, `first_entry_at`, `last_entry_at`,
   `total_entries`, with no `period` object).

This divergence does not affect runtime behaviour. It is tracked as a `spec` /
`tech-debt` GitHub issue and will be resolved by a `specification-manager` pass in
a future release.
