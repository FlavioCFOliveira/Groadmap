# Release Notes — Groadmap v1.9.0

**Date:** 2026-06-03
**Type:** Minor (feature)
**Tag:** `v1.9.0`
**Previous release:** `v1.8.2` (2026-06-03)

---

## Executive Summary

Groadmap v1.9.0 modernises the read-only `rmp web` interface and restructures its
navigation. The web UI is rebuilt on a shared Tabler-based layout, the
knowledge-graph viewer is reimplemented in D3.js (replacing the previous
Cytoscape.js renderer), and the single roadmap view is split into a dedicated
**Sprints** landing page and a separate **Tasks** page served from a new
`/roadmaps/{name}/tasks` endpoint. A new read-only sprint page at
`/roadmaps/{name}/sprints/{id}` shows one sprint and its member tasks. Authored
line breaks in sprint and task free-text are now preserved verbatim.

Every change in this release is additive or presentation-only. The `rmp` CLI
remains the sole write path, the read-only contract of the web interface is
unchanged, and every asset stays embedded and same-origin. There are no CLI
command-surface changes, no JSON output schema changes, no exit-code changes, and
no database schema changes. Under Semantic Versioning 2.0.0 this is a **MINOR**
release, fully backward compatible with `v1.8.2`. The database schema version is
unchanged at `1.6.0`, so existing installations require no migration.

---

## Highlights

- **Redesigned web UI:** a single shared Tabler-based layout replaces the
  per-page boilerplate across the index, roadmap, and graph pages.
- **D3.js knowledge-graph viewer:** the graph renderer is rebuilt on D3.js with a
  force-directed default and selectable D3 "Networks" gallery layouts (including a
  d3-sankey flow layout); the previous Cytoscape.js bundle is removed.
- **Sprints and Tasks split into separate pages:** the roadmap landing page is now
  the Sprints page, and the full task list moves to a new
  `/roadmaps/{name}/tasks` endpoint.
- **New read-only sprint page:** `/roadmaps/{name}/sprints/{id}` presents one
  sprint and its member tasks; an unknown or non-integer id returns HTTP `404`.
- **Authored line breaks preserved:** multi-line sprint descriptions and task
  long-text keep their newlines verbatim in the served HTML.
- **Self-contained and read-only by design:** every asset (Tabler, Tabler Icons,
  D3 with d3-sankey, the Inter web font, favicon) is embedded with `go:embed`; the
  interface renders fully offline and never writes data.
- No API, exit-code, JSON output, or database schema changes; schema version
  unchanged at `1.6.0`.

## Changes by Category

### Added — Dedicated Read-Only Sprint Page

A new read-only page at `/roadmaps/{name}/sprints/{id}` presents a single sprint
together with its member tasks. The sprint id is validated in the handler: an
unknown or non-integer id returns HTTP `404`, consistent with the roadmap-name
path guard already in place. The page answers `GET`/`HEAD` only; any other method
returns HTTP `405`. The behaviour is specified in `SPEC/WEB.md` (Routes and
Pages).

### Added — Separate Tasks Endpoint and Sprints Landing Page

The previous single roadmap view is split into two purpose-built pages:

| Route | Purpose |
|-------|---------|
| `/roadmaps/{name}` | Sprints landing page — three sprint tabs (Actual, Próximos, Concluídos), with the current/`OPEN` sprint expanded by default. |
| `/roadmaps/{name}/tasks` | Full task list with a per-row read-only detail modal. |

The sidebar now links **Sprints** to `/roadmaps/{name}` and **Tasks** to
`/roadmaps/{name}/tasks` (no in-page anchors). Both endpoints answer `GET`/`HEAD`
only and keep the existing roadmap-name `404` path guard. The data layer is split
to match: the Sprints page loads sprints with their ordered member tasks (without
a full task scan), and the Tasks page loads the full task list only. Each
Actual-tab task renders exactly one read-only detail modal.

### Changed — Read-Only Web UI Redesigned on a Shared Tabler Layout

The index, roadmap pages, and graph page now extend a single shared base layout —
the HTML skeleton, a Tabler-based navbar and shell, the favicon, and vendored CSS
and fonts — removing the per-page duplicated boilerplate that the previous design
carried. The vendored static-asset set is updated to match: the Cytoscape.js
bundle is removed and a new vendor bundle is embedded with `go:embed`, comprising
D3 (with d3-sankey), the Inter web font, the Tabler UI framework and Tabler Icons,
and the favicon. Each bundle's licence is recorded in
`internal/web/static/vendor/LICENSES.md`. The interface remains fully
self-contained: it renders offline and the server makes no outbound request.

### Changed — Knowledge-Graph Viewer Rebuilt on D3.js

The knowledge-graph renderer is reimplemented in D3.js, replacing the previous
Cytoscape.js implementation. A force-directed layout is the default, and the D3
"Networks" gallery layouts — including the d3-sankey flow layout — are selectable
from a dropdown. Graph data is fetched once and re-rendered in memory whenever the
layout changes, with no re-fetch, as specified in `SPEC/WEB.md` (FR7, AC10). The
graph store continues to be opened read-only; a web read never triggers a
checkpoint or write-ahead-log truncation.

### Changed — Authored Line Breaks Preserved in Sprint and Task Free-Text

Sprint descriptions and task long-text now retain their authored newlines instead
of collapsing them under HTML's default whitespace handling. A shared `pre-wrap`
stylesheet rule (`white-space: pre-wrap; word-break: break-word; overflow-wrap:
anywhere`) covers both the task detail modal and sprint descriptions across the
Sprints page and the sprint detail page. Output remains `html/template`-escaped;
no raw HTML is introduced, so the read-only and same-origin guarantees are
preserved.

### Changed — Project Governance

`CLAUDE.md` gains a "Core Working Principles" section that governs every
interaction — Ask-Never-Assume, Never-Guess, Measure-to-Decide,
Production-Grade-by-Default, Self-Contained Development, and the Specify ->
Implement -> Test -> Document workflow — reinforced across the Decision Matrix and
the Anti-Patterns sections. This is a documentation and process change only, with
no code or SPEC impact.

## Breaking Changes

None. The public CLI command surface, all JSON output schemas, exit codes, and
existing behaviour are fully backward compatible with `v1.8.2`. The web changes
are additive or presentation-only and preserve the read-only contract: the `rmp`
CLI remains the sole write path.

The only navigation change worth noting is that the full task list now lives at
`/roadmaps/{name}/tasks`; the roadmap landing page (`/roadmaps/{name}`) is now the
Sprints page. Both are reachable from the sidebar.

## Upgrade Notes

No migration steps are required. The web interface is read-only and adds no SQLite
migration; the database schema version is unchanged at `1.6.0`. The web interface
reads the same on-disk data the CLI already manages, in the same locations.

```sh
# Build from source
git fetch --tags
git checkout v1.9.0
go build -o ./bin/rmp ./cmd/rmp

# Verify
./bin/rmp --version
# Groadmap version 1.9.0

# Launch the read-only web interface (loopback by default)
./bin/rmp web
```

If you have bookmarked the roadmap task view, update it to `/roadmaps/{name}/tasks`;
the landing page `/roadmaps/{name}` now shows sprints.

## Benchmark Evidence

No performance-sensitive change was made to the existing roadmap, sprint, task,
audit, or graph code paths. This release is a read-only web presentation rework
plus additive read-only pages, governed by correctness and read-only-safety
contracts rather than by a throughput target; it is validated functionally by
`tests/test_35_web_interface.py` rather than by a throughput benchmark. The
benchmark baselines established in v1.3.0 remain the reference for the CLI data
paths. A dedicated web-server benchmark is deferred to a future release.

## Known Issues

One SPEC-vs-code divergence remains open and is unaffected by this release:

1. **`audit stats` JSON keys** — `SPEC/COMMANDS.md` documents one set of keys;
   the implementation emits a different (stable) set
   (`by_operation` / `by_entity_type` / `first_entry_at` / `last_entry_at` /
   `total_entries`, no `period` object).

This divergence does not affect runtime behaviour. It is tracked as a `spec` /
`tech-debt` GitHub issue and will be resolved by a `specification-manager` pass in
a future release.
