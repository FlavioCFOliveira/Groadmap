# Groadmap v1.3.0

- **Version:** 1.3.0
- **Date:** 2026-05-13
- **Release type:** Minor (backward compatible)
- **Previous tag:** v1.2.1
- **Comparison:** https://github.com/FlavioCFOliveira/Groadmap/compare/v1.2.1...v1.3.0

## Executive summary

Groadmap v1.3.0 is a quality-of-life and internal-quality release. The
command-line interface gains GNU-style `--flag=value` parsing and a complete
per-subcommand `--help` surface, so every command now has its own discoverable
help page that matches the runtime exactly. The internals have been
consolidated through a series of performance optimisations (recursive CTE for
transitive dependencies, single-query audit statistics, multi-row inserts,
deterministic SQL) and refactors that simplify the data-access layer. Test
coverage has been deepened in both Go unit tests and the Python E2E suite,
and the `SPEC/` directory has been reorganised for single-file
responsibility. No public CLI contract changed; this release is a drop-in
upgrade from `v1.2.1`.

## Highlights

### GNU-style `--flag=value` parsing

Every command now accepts both styles interchangeably:

```sh
rmp task create --title="Implement dashboard" --priority=P1
rmp task create --title  "Implement dashboard" --priority  P1
```

This brings `rmp` in line with the conventions used by `kubectl`, `gh`,
`docker` and most modern CLIs.

### Per-subcommand `--help`

`rmp <family> <subcommand> --help` now prints a dedicated, accurate help
block for every command in the `roadmap`, `task`, `sprint`, `audit` and
`backlog` families. The help texts:

- enumerate valid enum values for `status`, `priority` and `type`;
- list previously-omitted flags;
- distinguish similar listing commands (e.g. `task list` vs `backlog list`
  vs `sprint open-tasks`);
- describe the JSON output emitted on success;
- document conditional flags and workflow rules;
- list the exit codes that the command can return.

Help wording was audited and a final correction pass aligned every help
block with the actual runtime behaviour.

### Factual corrections in help output

Two dedicated fix passes (`fix(commands): correct factual errors in --help
output` and `fix(commands): correct factual errors and standardize wording
in --help`) brought help texts back into agreement with the implementation,
in particular around exit codes (`ExitInvalidData = 6`) and conditional
flag rules.

## Internal improvements

The release ships eight performance optimisations focused on database
access:

- `GetAuditStats` collapsed into a single `GROUP BY` query.
- `GetAuditEntries` query construction switched to `strings.Builder`.
- Task dependencies loaded via `group_concat`, removing the per-task N+1.
- `hasTransitiveDependency` switched from application-side BFS to a
  recursive Common Table Expression executed inside SQLite.
- Subtask and dependency completion guards execute as bulk queries.
- `AddTasksToSprint` uses a multi-row `INSERT`.
- `roadmapList` skips an unnecessary per-file `stat` by using
  `entry.Info()`.
- `ValidateIDString` uses `strconv.Atoi`.

Eight refactors clean the data-access layer:

- `AuditFilter` struct replaces a long positional argument list on
  `GetAuditEntries`.
- Position and swap mutations route through `WithTransaction`.
- Inline status string literals replaced with model-backed constants.
- Audit inserts consolidated into a single `LogAuditTx`.
- Unused `ConnectionCache` and `atexit` hooks removed.
- Retry wrapper around `sql.Open` dropped (the driver already handles it).
- `task_mutate` reuses the cached `db.Placeholders` lookup.
- `handleError` no longer falls back to string matching.

Two new helpers landed in `internal/utils`: `ValidateNumericRange` and
`ParseCommaSeparatedIDs`.

A whole-project style sweep renamed `interface{}` to the `any` alias.

The E2E suite itself expanded substantially during this cycle. New
coverage includes:

- backlog `list` and `show-next`;
- task dependency workflows end-to-end;
- subtask plus dependency completion guards and ordering;
- command-alias matrix (table-driven);
- `4096`-byte field-length boundary;

Ten weak `!= 0` exit-code assertions were replaced with explicit exit-code
and error-message checks. Go unit tests were also added for
`sprint open-tasks`.

## Documentation

The `SPEC/` tree was reorganised into single-responsibility files
optimised for LLM navigation. Versioning and change-history were removed
from the SPEC in favour of git as the source of truth. `SPEC/HELP.md` was
refreshed to match the per-subcommand help structure. `.claude/` agent
and skill references were updated after the project-local skill cleanup.

## Known issues

Two SPEC-vs-code divergences remain open and are tracked as GitHub
issues so a future `specification-manager` pass can decide how to
reconcile them. Neither divergence affects runtime behaviour:

1. **Exit-code mapping for `ErrInvalidInput`.**
   `SPEC/ARCHITECTURE.md` § *Error Reuse Policy / Sentinel Error
   Catalogue* states that `ErrInvalidInput` maps to exit code `2`. The
   implementation in `cmd/rmp/main.go` (`handleError`) maps
   `ErrInvalidInput`, `ErrValidation` and `ErrFieldTooLarge` to
   `ExitInvalidData = 6`, and the help texts shipped in this release
   reflect the implementation.

2. **`audit stats` JSON keys.**
   `SPEC/COMMANDS.md` § *Audit Statistics* documents the keys
   `operations_count`, `entity_type_count`, `first_entry`, `last_entry`
   and a `period.{since,until}` block. The implementation
   (`internal/models/audit.go` and `internal/commands/audit.go`) emits
   `by_operation`, `by_entity_type`, `first_entry_at`, `last_entry_at`
   and `total_entries`, with no `period` object.

Both divergences must be resolved by the `specification-manager` agent
(per the project policy that forbids direct SPEC edits) and will be
addressed in a follow-up release.

## Upgrade notes

No action required. v1.3.0 is fully backward compatible with v1.2.1.
Existing roadmap databases, JSON outputs, exit codes and command syntax
all continue to work unchanged. Users adopting the new `--flag=value`
syntax do not need to migrate their existing scripts.

To upgrade:

```sh
git fetch --tags
git checkout v1.3.0
make check
go build -o ./bin/ ./cmd/rmp
```

Or, for users who installed via `install.sh`, simply re-run the
installer against the new tag.
