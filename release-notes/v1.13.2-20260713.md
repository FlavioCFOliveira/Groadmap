# Release Notes — Groadmap v1.13.2

**Date:** 2026-07-13
**Type:** Patch (dependency refresh, toolchain security remediation, sprint `description` specification)
**Tag:** `v1.13.2`
**Previous release:** `v1.13.1` (2026-06-29)

---

## Executive Summary

Groadmap `v1.13.2` is a maintenance release. It refreshes every module
dependency, raises the required Go toolchain to **1.26.5** in order to remediate
a Go standard-library security advisory, and specifies the sprint `description`
field as the sprint's macro goal across every surface that states it.

The headline dependency change is the embedded **GoGraph** knowledge-graph
engine, which moves from `v0.6.0` to `v0.7.0`. That upgrade is not cosmetic: it
carries two write-correctness fixes that reach `rmp` users directly through the
`rmp graph` command family, and a broad denial-of-service hardening pass across
the graph store.

No `rmp` source behaviour changes in this release. Every user-visible fix in it
is delivered either by the upgraded engine or by the corrected help and
specification text.

---

## Highlights

- **Toolchain security remediation.** The Go floor moves to `1.26.5`, which
  carries the fix for **GO-2026-5856**, an Encrypted Client Hello privacy leak in
  `crypto/tls`. This is a toolchain vulnerability that no dependency change can
  remediate.
- **`MERGE` now behaves correctly.** The single most common Cypher
  graph-building idiom no longer applies partially when a pattern has a fresh
  endpoint.
- **No more silent write loss on parallel edges.** A parallel edge on a
  non-multigraph engine is now rejected with a typed error instead of being
  silently discarded.
- **Sprint `description` has a stated purpose.** The field is now specified,
  everywhere it is documented, as the macro goal of the development effort the
  sprint delivers, so that a caller cannot read the help and conclude that any
  free text will do.

## Changes by Category

### Security — Go Floor Raised to 1.26.5 (GO-2026-5856)

The Go standard library's `crypto/tls` is affected by **GO-2026-5856**, an
Encrypted Client Hello privacy leak, in Go 1.26.4 and earlier. The fix ships in
Go 1.26.5.

This is a **toolchain** vulnerability, not a module vulnerability: it is
remediated by the toolchain version alone, and no dependency change can
remediate it. Accordingly, the `go` directive in `go.mod` now declares
`go 1.26.5`, and Groadmap MUST NOT be built or released with an older toolchain.
The CI and release workflows resolve their Go version from `go.mod` via
`go-version-file`, so they track the floor automatically with no workflow edit.

`SPEC/BUILD.md` is now the authoritative statement of the required Go version and
records the two independent constraints that set it: the **minor** floor (Go
1.26), set by the GoGraph dependency, and the **patch** floor (Go 1.26.5), set by
this security requirement.

### Fixed — Graph Write Correctness (via GoGraph v0.7.0)

- **`MERGE` applies whole-pattern match-or-create semantics.** A `MERGE` of a
  pattern with a fresh endpoint was previously silently incomplete. The write now
  either fully applies or is rejected with a typed error — never a partial,
  silent result. This affects every `rmp graph update` and `rmp graph create`
  query that uses `MERGE`.
- **Parallel edges are rejected, not silently dropped.** On a non-multigraph
  engine — the documented default configuration — writing a parallel edge
  previously discarded the write with no error. It is now rejected with a typed
  error, closing a genuine write-loss gap.

Both are Consistency fixes in the ACID sense: a write that used to vanish
silently, or apply only in part, now either fully applies or fails loudly.

### Fixed — AI Agent Contract: `sprint update` Summary

The machine-readable AI Agent Contract (`rmp --ai-help`) described
`sprint update` as editing "title, description or capacity cap". It omitted the
execution-order field, which the subcommand has accepted since `v1.10.0`. The
summary and description now state all four editable fields: title, description,
capacity cap, and execution order.

### Changed — Sprint `description` Specified as the Macro Goal

The `description` field of a sprint is now specified, on every surface that
states it, as the **high-level (macro) goal of the development effort the sprint
delivers**: a new development, a fix, a refactoring, or another kind of change.

Together with the sprint title, the description must give a human reader or an AI
agent a clear macro idea of what the sprint's tasks are specifically aimed at. It
states the macro goal **only**: detailed scope, technical detail, and acceptance
conditions belong to the sprint's tasks, which carry them in full through their
functional requirements, technical requirements, and acceptance criteria. A label
such as `"Sprint 3"`, or a restatement of the title, no longer satisfies the
documented contract.

The semantics are now stated in:

- the plain-text help for `sprint create` and `sprint update`;
- the AI Agent Contract (`rmp --ai-help`), in the `flags[]` entry for
  `--description` on both subcommands;
- `SPEC/MODELS.md` (the canonical definition, under "Sprint Field Constraints"),
  `SPEC/COMMANDS.md`, `SPEC/HELP.md`, and `SPEC/README.md`;
- `DOCS/commands/sprint.md` and the `README.md` examples.

This is a documentation and specification change. The field was **already**
mandatory on `sprint create`, and no validation behaviour changes: `SPEC/MODELS.md`
now records the field as required (NOT NULL), which reconciles the specification
with what the code already enforced.

Beyond the two write fixes described above, **GoGraph `v0.7.0`** lands a security
and denial-of-service hardening pass across the graph store: a size-bounded
manifest, symlink rejection on every file the engine opens by name (including the
write-ahead log and snapshots), bounded allocation of frame payloads, and memory
ceilings in the Cypher engine.

It also adds eight openCypher builtin functions — `elementId`, `timestamp`,
`randomUUID`, `isNaN`, and the `toStringList` / `toIntegerList` / `toFloatList` /
`toBooleanList` family — which become available to `rmp graph query`, since `rmp`
passes Cypher through to the engine.

The upgrade is a pre-1.0 additive change: no exported Go API is broken, and the
on-disk graph store requires no migration. The pin is reconciled to `v0.7.0` in
`SPEC/BUILD.md`, `SPEC/GRAPH.md`, and `SPEC/ARCHITECTURE.md`.

### Changed — Version State Removed From `SPEC/VERSION.md`

`SPEC/VERSION.md` no longer carries version state. The `Current Version` table is
removed, the illustrative constants are genericised to `X.Y.Z`, and the documented
release process is aligned with the process the project actually follows.

Git tags and `git log` are now the single source of truth for the history of the
specification, in line with the project's versioning policy. Correspondingly,
`SPEC/BUILD.md` becomes the authoritative statement of the required Go version,
and the other specification files point to it rather than restate it.

### Internal

- **Help-content unit tests.** New unit tests cover the AI Agent Contract
  generator (`internal/aihelp`) and the command help content
  (`internal/commands`), pinning the sprint `description` semantics in both the
  generated contract and the plain-text help.
- **End-to-end contract coverage extended.** The help and exit-code contract
  suite gains two cases asserting that the macro-goal semantics are stated both
  in the plain-text help and in `rmp --ai-help`.
- **Project governance.** `CLAUDE.md` is revised to tighten the internal working
  rules. This affects contributors only; it has no effect on the product.

## Breaking Changes

**None.** This release contains backward-compatible fixes, dependency
maintenance, and documentation only.

No `rmp` command, flag, or JSON success schema is added, removed, or renamed; no
exit code changes; the GoGraph upgrade is additive with no exported API break and
no on-disk store migration; and the database schema version is unchanged
(`1.8.0`).

### A Note on the Go Floor and Semantic Versioning

Raising a minimum toolchain version is a stricter *build* requirement, so it is
worth stating explicitly why this remains a **PATCH** release rather than a MINOR
one:

1. **The Go minor floor does not move.** Go 1.26 has been required since
   `v1.7.0`. Only the *patch* component moves, 1.26.4 to 1.26.5 — the same
   language version, within Go's compatibility promise. No language feature is
   gated by it.
2. **Groadmap exposes no importable Go API.** Every package other than `cmd/rmp`
   (package `main`) lives under `internal/`, which Go structurally forbids
   external modules from importing. The convention of treating a raised Go floor
   as a MINOR exists to protect *importing* modules; Groadmap has none by
   construction, so no downstream build can break.
3. **Consumers are unaffected in practice.** Released binaries are distributed
   prebuilt for seven targets, and need no Go toolchain at all. For
   `go install`, `GOTOOLCHAIN=auto` (the default since Go 1.21) transparently
   fetches the required toolchain.
4. **It is a security fix.** Under `SPEC/VERSION.md`, a backward-compatible fix
   is precisely a PATCH.

## Upgrade Notes

No manual migration steps are required. This release adds no SQLite migration of
its own, the database schema version is unchanged, and the GoGraph upgrade needs
no on-disk store migration.

Building from source now requires **Go 1.26.5 or later**. If you build with
`GOTOOLCHAIN=local` and are pinned to an older Go, update your toolchain first;
with the default `GOTOOLCHAIN=auto`, the Go command fetches it for you.

```sh
# Build from source (requires Go 1.26.5+)
git fetch --tags
git checkout v1.13.2
go build -o ./bin/rmp ./cmd/rmp

# Verify
./bin/rmp --version
# Groadmap version 1.13.2

./bin/rmp --ai-help | grep binary_version
#   "binary_version": "1.13.2",
```

If you author sprints, note that `-d, --description` is expected to carry the
sprint's macro goal. Existing sprints are unaffected; nothing is revalidated or
rewritten.

```sh
# Before: a label, which no longer satisfies the documented contract
rmp sprint create -r myproject -t "Auth hardening" -d "Sprint 3"

# After: the macro goal of the effort the sprint delivers
rmp sprint create -r myproject -t "Auth hardening" \
  -d "Deliver session-based authentication for every write command."
```

## Benchmark Evidence

This release refreshes dependencies, raises the Go floor, and corrects
documentation; it does not change the hot data path of any benchmarked `rmp`
command. The Groadmap CLI command benchmarks therefore remain the reference
established in `v1.3.0` and are not re-measured here. Correctness is governed by
the unit, coverage, and E2E suites above rather than by a throughput target.

The GoGraph `v0.7.0` hardening pass introduces bounded allocations and memory
ceilings in the graph store. These are safety limits well above the working-set
size of a roadmap knowledge graph, and no change in `rmp graph` latency was
observed across the E2E suite.
