# Release Notes — Groadmap v1.8.0

**Date:** 2026-06-03
**Type:** Minor (feature)
**Tag:** `v1.8.0`
**Previous release:** `v1.7.0` (2026-06-02)

---

## Executive Summary

Groadmap v1.8.0 introduces the **`rmp web`** command: a read-only,
self-contained, mobile-first web interface for browsing every roadmap under
`~/.roadmaps/`, its tasks and sprints, and an interactive knowledge-graph
visualisation. The interface runs an HTTP server embedded in the `rmp` binary,
serves server-rendered HTML from `go:embed`ded assets, and renders fully
offline. The `rmp` CLI remains the sole write path; the web interface never
creates, edits, or deletes data.

This release also aligns the CLI exit-code mapping with the canonical contract
in `SPEC/ARCHITECTURE.md`. `ErrInvalidInput` now maps to exit `2` (misuse),
while value, range, enum, date, state-transition, and business-rule validations
are reclassified to `ErrValidation` so they remain exit `6` (invalid data). This
resolves the first "Known Issues" item carried since `v1.3.0`.

Both changes are additive or corrective. No JSON output schema changes, and the
database schema version is unchanged at `1.6.0` because the web interface is
read-only and adds no SQLite migration. Under Semantic Versioning this is a
**MINOR** release, fully backward compatible with `v1.7.0`. Existing
installations require no migration.

## Highlights

- New read-only `rmp web` command: embedded HTTP server, server-rendered pages,
  and an interactive Cytoscape.js knowledge-graph visualisation.
- Fully self-contained: every asset (HTML, CSS, client JavaScript, the vendored
  Cytoscape.js library, favicon) is embedded with `go:embed`; the interface
  renders offline and the server makes no outbound request.
- Read-only by design: no route creates, edits, or deletes data; the graph store

  write-ahead-log truncation.
- Loopback by default (`127.0.0.1:8787`), with explicit opt-in for non-loopback
  binds and an ephemeral-port fallback when the default port is in use.
- Exit-code mapping aligned with `SPEC/ARCHITECTURE.md`, resolving a long-standing
  SPEC-vs-code divergence.
- One new end-to-end test suite (`tests/test_35_web_interface.py`); full E2E
  suite at 24/24.

## Changes by Category

### Features — The `rmp web` Command

`rmp web` starts an HTTP server embedded in the `rmp` binary, opens a local
browser to it, and lets the user navigate the roadmaps found under
`~/.roadmaps/`. The interface presents data; it never changes it. The command is
specified in `SPEC/WEB.md`.

**Routes and pages.** The server exposes three kinds of page plus a JSON data
endpoint and embedded static assets:

| Route | Purpose |
|-------|---------|
| `/` | Roadmap index — lists every roadmap under `~/.roadmaps/`. |
| `/roadmaps/{name}` | Roadmap detail — tasks and sprints for one roadmap. |
| `/roadmaps/{name}/graph` | Knowledge-graph page with the interactive visualisation. |
| `/roadmaps/{name}/graph/data` | JSON graph-data endpoint consumed by the page. |
| `/static/...` | Embedded static assets (CSS, JavaScript, favicon). |

Routes answer `GET` and `HEAD` only; any other method returns HTTP `405`. The
roadmap `{name}` from the URL is validated before any filesystem path is built;
an invalid or unknown name returns HTTP `404`. The server is built only from
Go's standard library (`net/http`).

**Self-contained deliverable.** Every component required to render and operate
the interface — HTML templates, the stylesheet, all client JavaScript including
the vendored Cytoscape.js graph library, and the favicon — is embedded into the
binary at build time with `go:embed`. The interface requires no external runtime
dependency, no JavaScript build toolchain, no `node_modules`, and no content
delivery network. It renders and functions fully offline with only the single
`rmp` binary present on disk.

**Read-only guarantee.** The interface exposes no route that creates, edits, or
deletes data. Tasks and sprints are read from each roadmap's SQLite `project.db`
and the knowledge graph is read from the GoGraph store under
`~/.roadmaps/<name>/graph/`, both opened read-only. A web read never triggers a
graph checkpoint or write-ahead-log truncation. The `rmp` CLI remains the sole
write path for all roadmap data.

**Responsive and mobile-first.** Base styles target small phone-sized viewports
first and progressively enhance for larger viewports. The layout adapts fluidly
across viewport sizes on every page, including the interactive knowledge-graph
visualisation.

**Bind address and lifecycle.** The server binds `127.0.0.1:8787` by default; a
non-loopback bind (for example `--host 0.0.0.0`) is an explicit opt-in. When
`--port` is omitted and the default port is already in use, the server falls
back to an OS-chosen ephemeral port so it still starts. The process is
long-lived: `SIGINT`/`SIGTERM` shut it down gracefully with exit `0`.

**Command surface.** `rmp web` accepts the flags `--host`, `--port`,
`--no-open`, and `-h`/`--help`. It is the one command exempt from the
always-required-roadmap rule: it accepts no `-r`/`--roadmap` flag and no
subcommands.

### Fixed — Exit-Code Mapping Aligned with `SPEC/ARCHITECTURE.md`

`ErrInvalidInput` now maps to exit `2` (misuse: unknown flags and subcommands,
malformed or non-numeric IDs), while value, range, enum, date, state-transition,
and business-rule validations are reclassified to `ErrValidation` so they remain
exit `6` (invalid data). This brings the implementation in `cmd/rmp/main.go` into
line with the contract already documented in `SPEC/ARCHITECTURE.md`, resolving
the first item recorded under "Known Issues" since `v1.3.0` (the
`ErrInvalidInput` exit-code divergence). The change affects only error paths the
SPEC already defined; no JSON output schema changes.

## Breaking Changes

None. The public CLI command surface, all JSON output schemas, and existing
behaviour are fully backward compatible with `v1.7.0`. The `rmp web` command is
purely additive. The exit-code change is corrective and affects only the exit
status returned on validation/misuse error paths, bringing the binary into line
with the documented contract.

## Upgrade Notes

No migration steps are required. The web interface is read-only and adds no
SQLite migration; the database schema version is unchanged at `1.6.0`. The web
interface reads the same on-disk data the CLI already manages, in the same
locations.

```sh
# Build from source
git fetch --tags
git checkout v1.8.0
go build -o ./bin/rmp ./cmd/rmp

# Verify
./bin/rmp --version
# Groadmap version 1.8.0

# Launch the read-only web interface (loopback by default)
./bin/rmp web
```

If you rely on the process exit status of `rmp` in scripts or CI, note that
misuse errors (`ErrInvalidInput`) now exit with `2` rather than `6`; value,
range, enum, date, state, and business-rule validation errors continue to exit
with `6`.

## Benchmark Evidence

No performance-sensitive change was made to the existing roadmap, sprint, task,
audit, or graph code paths. The benchmark baselines established in v1.3.0 remain
the reference for those paths. The new `rmp web` code path is a read-only
presentation layer governed by correctness and read-only-safety contracts rather
than by a throughput target; it is validated functionally by
`tests/test_35_web_interface.py` rather than by a throughput benchmark. A
dedicated web-server benchmark is deferred to a future release.

## Known Issues

One SPEC-vs-code divergence remains open and is unaffected by this release:

1. **`audit stats` JSON keys** — `SPEC/COMMANDS.md` documents one set of keys;
   the implementation emits a different (stable) set
   (`by_operation` / `by_entity_type` / `first_entry_at` / `last_entry_at` /
   `total_entries`, no `period` object).

This divergence does not affect runtime behaviour. It is tracked as a `spec` /
`tech-debt` GitHub issue and will be resolved by a `specification-manager` pass
in a future release.

The previously listed exit-code divergence for `ErrInvalidInput` is **resolved**
in this release (see the "Fixed" section above).
