## Detailed Findings

### GOSEC-001: Dynamic SQL Field Name Construction

**Severity:** LOW
**CVSS v3.1:** CVSS:3.1/AV:L/AC:H/PR:H/UI:R/S:U/C:N/I:L/A:N (2.0)
**CWE:** CWE-89 (SQL Injection) - Partial

**Location:**
- `internal/commands/task.go:523-528` (taskEdit function)
- `internal/db/queries.go:257-263` (UpdateTask function)
- `internal/db/queries.go:343-345` (UpdateTaskStruct function)

**Description:**
SQL queries are constructed using `fmt.Sprintf` with field names concatenated directly into the query string. While field names are validated against a whitelist before use, this pattern could become vulnerable if the whitelist validation is bypassed or modified.

**Code Snippet:**
```go
// From internal/commands/task.go:523-528
for field, value := range updates {
    setParts = append(setParts, fmt.Sprintf("%s = ?", field))
    args = append(args, value)
}
query := fmt.Sprintf("UPDATE tasks SET %s WHERE id = ?", strings.Join(setParts, ", "))
```

**Mitigation Factors:**
1. Field names are validated against `allowedTaskUpdateFields` whitelist
2. Values are properly parameterized with `?` placeholders
3. User cannot directly control field names - only through predefined flag mappings

**Remediation Recommendation:**
```go
// Use a field mapping instead of direct concatenation
var fieldMapping = map[string]string{
    "title": "title",
    "functional_requirements": "functional_requirements",
    // ... etc
}

// Then use the mapped value
if dbField, ok := fieldMapping[field]; ok {
    setParts = append(setParts, dbField + " = ?")
}
```

**Risk Acceptance:**
Current implementation is acceptable given the whitelist validation and internal-only usage. Risk is minimal as this is a CLI application with local SQLite database.

### GOSEC-002: JSON HTML Escaping Disabled

**Severity:** INFORMATIONAL
**CWE:** CWE-80 (Improper Neutralization of Script-Related HTML Tags)

**Location:**
- `internal/utils/json.go:40` (PrintJSON function)
- `internal/utils/json.go:54` (PrintJSONIndent function)

**Description:**
HTML escaping is intentionally disabled in JSON output via `encoder.SetEscapeHTML(false)`. This is documented in code comments as intentional for CLI use.

**Security Note:**
- CLI output goes to stdout, not a web browser
- No web interface exists in this application
- If output is consumed by web applications, HTML injection becomes possible

**Current Mitigation:**
Code comments clearly document this decision and its rationale.

**Recommendation:**
No immediate action required. If future web interface is added, ensure proper output encoding at the web layer.
